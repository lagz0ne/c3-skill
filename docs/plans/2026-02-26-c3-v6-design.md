# C3 v6 Design: Unified Skill + Embedded Go CLI

**Date:** 2026-02-26
**Status:** Draft
**Target version:** 6.0.0

## Problem

v5 has distribution friction:
- 6 skills + 6 agents = complex routing, unreliable skill triggering
- CLI distributed via npm (`@lagz0ne/c3x`) adds a dependency
- References duplicated across skill directories, synced via build script
- Templates live in plugin, read by Claude at runtime

Users want: install one thing, it works.

## Decision

Consolidate into **one self-contained skill** with an **embedded Go CLI**.

## Architecture

### What Changes

| Aspect | v5 | v6 |
|--------|----|----|
| Skills | 6 separate | 1: `/c3` |
| Agents | 6 named agents | 0 (anonymous subagents via Task when needed) |
| CLI | TypeScript, npm package | Go binary, embedded in skill |
| Distribution | npm + Claude Code plugin | Plugin only (self-contained) |
| References | Duplicated per skill | Single copy in `skills/c3/references/` |
| Templates | In plugin, Read by Claude | Embedded in Go binary via `go:embed` |
| Invocation | `npx -y @lagz0ne/c3x <cmd>` | `bash <skill-dir>/bin/c3x.sh <cmd>` |
| Routing | Skill descriptions compete | CLAUDE.md injection during onboard |

## Go CLI

### Commands (4)

| Command | Purpose | Key flags |
|---------|---------|-----------|
| `c3x init` | Scaffold `.c3/` with embedded templates | — |
| `c3x list` | Topology: entities, relationships, frontmatter | `--json`, `--flat` |
| `c3x check` | Structural validation: broken links, orphans, duplicates | `--json` |
| `c3x add <type> <slug>` | Create entity with auto-numbering + wiring | `--container`, `--feature` |

`sync` command dropped (guard skills are a separate concern).

### Design Principles

- **SMALL, TINY** — single static binary, no runtime deps
- **Mechanical only** — parsing, validation, scaffolding. No search, no reasoning.
- **Templates embedded** — `go:embed` bundles .md templates into the binary
- **Fast** — Go cold start is <10ms vs npx's ~2s download

### Shell Wrapper (`c3x.sh`)

```bash
#!/usr/bin/env bash
# Detect OS/arch, run the right binary
SCRIPT_DIR="$(cd "$(dirname "$0")" && pwd)"
OS=$(uname -s | tr '[:upper:]' '[:lower:]')
ARCH=$(uname -m)
case "$ARCH" in
  x86_64) ARCH="amd64" ;;
  aarch64|arm64) ARCH="arm64" ;;
esac
exec "$SCRIPT_DIR/c3x-${OS}-${ARCH}" "$@"
```

## The `/c3` Skill

### SKILL.md Style

- **Telegram style** — bulletpoint, no prose, sacrifice grammar for brevity
- No instructive text — cleaned instructions only
- Tables > paragraphs
- Extreme focus per section

### SKILL.md Structure (~300-400 lines)

```
Frontmatter → CLI setup → Intent table → Precondition → Shared rules → Per-op summaries
```

### SKILL.md Sketch

```markdown
## CLI

Binary: `bin/c3x.sh` (relative to this skill dir)
All calls: `bash <skill-dir>/bin/c3x.sh <command> [args]`

## Intent Classification

| Keywords | Op | Ref |
|----------|----|---------|
| adopt, init, scaffold, bootstrap | onboard | references/onboard.md |
| where, explain, how, diagram, trace | query | references/query.md |
| audit, validate, check docs, drift | audit | references/audit.md |
| add, change, fix, implement, refactor | change | references/change.md |
| pattern, convention, ref, "list refs" | ref | references/ref.md |
| impact, "what breaks", assess, sweep | sweep | references/sweep.md |

## Dispatch

- Analyze request → classify
- Ambiguous → AskUserQuestion w/ 6 ops
- Load `references/<op>.md`
- Execute
- Subagents: use Task tool for parallelism when reasoning suggests it helps

## Precondition

- `c3x.sh list --json` → fails? → route to onboard
- Exception: onboard doesn't need existing .c3/

## Operations

### onboard
- Pre: no .c3/ exists
- `c3x init` → scaffold
- Discover: containers, components, refs (Socratic, AskUserQuestion)
- Gate each stage
- Post: inject CLAUDE.md routing, show capabilities overview

### query
- `c3x list --json` → match entity → Read doc → explore code
- Subagents: parallel container exploration when multi-scope

### audit
- `c3x check` → structural issues
- `c3x list --json` → inventory
- Phases 2-10: semantic validation (Read+Grep+reasoning)

### change
- `c3x list --json` → impact analysis
- ADR: `c3x add adr <slug>` → fill content → user approves
- Execute: `c3x add component/ref` for scaffolding, implement code
- Audit: `c3x check` + verify docs match code

### ref
- Add: `c3x add ref <slug>` → fill → discover usage → update citings
- Update: find citings → check compliance → surface impact
- List: `c3x list --json` → filter type=ref

### sweep
- `c3x list --json` → identify affected entities
- Subagents: parallel per-container assessment
- Synthesize: constraints, risks, recommendations
- Advisory only — route to change for implementation
```

### Onboard: Capabilities Reveal

After scaffold, onboard presents what `/c3` can do:

```
## Your C3 toolkit is ready

| Command | What it does |
|---------|-------------|
| `/c3` query | Ask about architecture — "where is auth?", "how does X work?" |
| `/c3` audit | Validate docs — drift, broken links, stale refs |
| `/c3` change | Modify architecture — add components, refactor, implement |
| `/c3` ref | Manage patterns — create/update/list cross-cutting conventions |
| `/c3` sweep | Impact assessment — "what breaks if I change X?" |

Just say `/c3` + what you want. It'll figure out the rest.
```

Also injects into CLAUDE.md:

```markdown
# Architecture
This project uses C3 docs in `.c3/`.
For architecture questions, changes, audits → `/c3`.
Operations: query, audit, change, ref, sweep.
```

### Routing Reliability

- Skill description (1024 char limit) = backup only
- Primary routing: CLAUDE.md injection during onboard
- CLAUDE.md always in context → reliable `/c3` triggering

### Subagent Strategy

- No named agents
- Anonymous subagents via Task tool when parallelism helps
- Skill reasons about when: multi-container queries, independent implementation tasks, parallel entity assessment
- Not prescribed — decided per-request

## Build & Release

### Build Script (`scripts/build.sh`)

```bash
# 1. Cross-compile Go CLI for all targets
cd cli/
for target in linux-amd64 linux-arm64 darwin-amd64 darwin-arm64; do
  GOOS=${target%-*} GOARCH=${target#*-} go build -o ../skills/c3/bin/c3x-${target} .
done

# 2. Copy shell wrapper
cp scripts/c3x.sh skills/c3/bin/c3x.sh

# 3. Plugin is now self-contained under skills/c3/
```

### Release Pipeline

1. Push to main with version bump in `VERSION`
2. GitHub Actions:
   - Cross-compile Go CLI
   - Assemble plugin (skills/c3/ with binaries)
   - Create GitHub release with plugin zip
3. No npm publish step

### Plugin Installation

```bash
# From marketplace
claude plugin install c3-skill

# From GitHub
claude plugin install lagz0ne/c3-skill
```

## Migration Path

### For existing v5 users

1. Plugin auto-updates (marketplace handles versioning)
2. Old skills (c3-onboard, c3-query, etc.) replaced by single `/c3`
3. Existing `.c3/` directories are 100% compatible (same format)
4. CLAUDE.md routing section added on first `/c3` invocation
5. `npx -y @lagz0ne/c3x` calls in existing CLAUDE.md files need updating to use the binary path

### Breaking changes

- All 6 individual skill names gone (`c3-onboard`, `c3-query`, etc.)
- All 6 agent names gone (`c3-lead`, `c3-navigator`, etc.)
- npm package `@lagz0ne/c3x` deprecated
- Skill invocation: `/c3` instead of individual skill names
