# Auth and Access

## Goal

Trace the authentication and authorization path from login through permission checks in flows.

## Narrative

Authentication and authorization span three layers: HTTP session (cookies),
application RBAC, and NATS transport auth.

**Authentication** (ref-authentication): Google OAuth for production, test
token bypass for E2E. Both paths require the user to already exist in the
database — OAuth alone doesn't grant access. Session is a cookie with
Base64-encoded email, HttpOnly. No refresh tokens stored (stateless).

**RBAC** (ref-rbac): JSON permissions per role with parent inheritance.
Special `owner` role checked via `rbacQueries.isOwner()`. Effective
permissions = parent merged with role. Built-in roles: owner, finance,
admin, bod. All mutations logged to `security_events`.

**UserActor** (c3-202): The `currentUserTag` carries the full actor —
email, team, capabilities, permissions, and `can()`/`asserts()` methods.
Set by `getCurrentUserMiddleware`, consumed by flows via `seekTag()`.

**NATS auth** (ref-nats-jwt-auth): JWT resolver for WebSocket connections.
Credentials generated by c3-209. Separate from HTTP auth — NATS has its
own identity layer.

## Access Check Pattern

```typescript
const currentUser = ctx.data.seekTag(currentUserTag);
currentUser.asserts('some.permission'); // throws if denied
```

## Edge Cases

No roles = no permissions. Role expiry treated as unassigned.
Deleting the last owner should be prevented. Self role removal blocked
for last owner.
